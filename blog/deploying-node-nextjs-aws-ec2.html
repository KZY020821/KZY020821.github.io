From Localhost to the Cloud: Deploying Full-Stack Next.js on AWS EC2

A story of overcoming memory limits, version conflicts, and networking puzzles to host a production-ready application on a single micro instance.

Next.js 16 AWS EC2 Node.js Infrastructure

The "It Works on My Machine" Moment

We've all been there. The CRUD application is humming along perfectly on localhost:3000. The frontend is snappy, the backend is responsive, and every API call returns a satisfying 200 OK. But as every developer eventually learns, the real test begins when you try to move that code from the comfort of your MacBook to the wild frontier of a raw cloud server.

Today, I embarked on a journey to deploy a full-stack Next.js 16 and Node.js application onto a single AWS EC2 t2.micro instance. What seemed like a straightforward task quickly turned into a masterclass in resource management and cloud networking.

The First Wall: The Version Gap

After SSH-ing into the EC2 instance, I was greeted by the first hurdle: Node.js 18. While stable, modern frameworks like Next.js 16 demand at least v20.9.0. Instead of fighting the system package manager, I turned to NVM (Node Version Manager).

Upgrading the Environment
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.7/install.sh | bash
source ~/.bashrc
nvm install 20
node -v # Success: v20.x.x

The Second Wall: The Memory Crash

With the correct Node version, I ran npm run build. Five minutes in, the terminal froze. "Broken pipe." The connection dropped.

The culprit? Next.js 16's build process is a resource hog, and the 1GB of RAM on a free-tier EC2 instance simply wasn't enough. The server was literally choking to death. To fix this, I had to trick the server into thinking it had more memory by creating a Swap File.

Creating Virtual RAM
# Create a 2GB swap file from thin air
sudo dd if=/dev/zero of=/swapfile bs=128M count=16
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile

With this "virtual breathing room," the build process completed successfully.

The Final Puzzle: Networking & Security

The app was built, but it was invisible to the world. I had to orchestrate two separate processes running in the background:

  • Backend: Express on Port 3000
  • Frontend: Next.js on Port 3001

I used nohup to ensure they kept running even after I disconnected. But even then, the browser showed "Failed to connect." This led me to the AWS Security Group dashboard, where I had to manually punch holes in the firewall to allow traffic through those specific ports.

The Architecture of Success

3000 API Port
3001 Web Port
2GB Swap Added

The result? A fully functional, production-ready environment running on the humblest of cloud hardware. This journey reminded me that deployment isn't just about moving files—it's about understanding the environment and the constraints of the hardware you're working with.
