From Localhost to AWS: Building Clair Tax — and How Claude 10x'd My Productivity

A deep dive into building a Malaysian tax platform as a full AWS monorepo — and how AI pair programming changed what's possible for a solo developer.

Next.js Spring Boot FastAPI AWS Claude AI OCR

The Problem Worth Solving

Every year, millions of Malaysians file their income tax returns through LHDN (Lembaga Hasil Dalam Negeri). The process is painful. There are over 12 relief categories — medical expenses, lifestyle purchases, education fees, EPF contributions, life insurance, and more — each with their own annual caps and eligibility rules.

Most people handle this by hoarding physical receipts in a shoebox, frantically categorising them in March, and making rough guesses about how much relief they can claim. The result is either leaving money on the table or overclaiming and risking an audit.

The idea behind Clair Tax was simple: scan your receipts throughout the year, let the system categorise them, and get a live tax estimate at any point. Receipts in, tax position out.

What started as a weekend side project grew into a proper full-stack system with a 3-service monorepo deployed on AWS. This post covers the architecture, the deployment gaps that surprised me, and — the part I'm most excited to share — how Claude AI fundamentally changed the velocity of what a solo developer can ship.

Choosing the Stack — And Why

Every framework choice in Clair Tax was deliberate. Here's the reasoning behind each one.

  • Next.js (Frontend): The tax dashboard needs to display a real-time tax estimate that updates as receipts are processed. SSR matters here — when a user shares their dashboard link, the estimate should be pre-rendered on the server, not a blank loading spinner.
  • Spring Boot (Backend API): I chose Spring Boot over NestJS for two reasons. First, the Java ecosystem's JPA/Hibernate support for complex relational data (tax reliefs have a deeply nested parent-child structure) is mature and well-documented. Second, frankly: Spring Boot appears on more Malaysian enterprise job postings than any Node.js framework, and building with it deepens a skill set that's directly marketable.
  • FastAPI (AI Service): OCR is inherently I/O-bound and compute-heavy. FastAPI's async-native design lets you run multiple image-processing tasks concurrently without blocking the event loop. Pydantic's strict type validation is invaluable when your input (OCR output) is unpredictable — amounts can be strings, floats, or malformed entirely.
  • PostgreSQL on RDS: Tax data is relational: one user has many receipts, each receipt has many line items, each line item maps to a relief category with year-specific caps. A document store would require application-level joins. PostgreSQL on RDS gives you managed backups, automated patching, and Multi-AZ failover without running your own database server.
  • SQS + Lambda: The OCR pipeline takes 2–3 seconds per receipt. If you process that synchronously inside the HTTP request cycle, every receipt upload blocks a Spring Boot thread for 3 seconds. On a t3.medium with Tomcat's default 200 threads, you'd exhaust the thread pool under moderate load. SQS decouples the upload from the processing — the API returns in under 200ms, and Lambda handles the heavy lifting asynchronously.

A few things worth calling out. The browser uploads directly to S3 via a pre-signed URL — the image never passes through EC2, which keeps the instance's network I/O low and eliminates a potential bottleneck. The Lambda function is the only component that talks to FastAPI, meaning the OCR service is never exposed to the public internet.

Local Development to AWS: The Painful Gaps

Getting this running locally is one thing. Getting it running correctly on AWS is another. Here are the gaps that cost me the most time.

Docker Compose + LocalStack

In development, I used LocalStack to emulate AWS SQS and S3 locally. This lets you run aws sqs send-message and aws s3 cp against localhost:4566 without incurring AWS costs or needing a live AWS environment. The catch: LocalStack doesn't emulate Lambda triggers, so I ran the Lambda consumer as a standalone polling script locally.

Secrets Management

Spring Boot's application-prod.properties file holds the RDS connection string, credentials, and SQS queue URLs. In production, these values are fetched at startup from AWS Secrets Manager using the Spring Cloud AWS Secrets Manager integration. Never hardcode credentials — not even in a private repo.

EC2 Bootstrap

The EC2 instance uses a user data script that runs on first launch. It installs Java 21, Node 20, PM2 (for keeping Next.js alive), and pulls the latest monorepo build from S3. PM2 handles automatic restarts if either process crashes.

Security Groups — The Gotcha

This tripped me up early. My initial Security Group allowed port 8080 (Spring Boot) open to 0.0.0.0/0. That means anyone on the internet could call your API directly, bypassing the ALB and any WAF rules you've set up. The correct rule: port 8080 should only accept inbound traffic from the ALB's Security Group ID — not from the world. (More on how Claude caught this below.)

IAM Roles

Two IAM roles are needed. The EC2 instance role gets sqs:SendMessage and s3:PutObject (to enqueue jobs and generate pre-signed upload URLs). The Lambda execution role gets sqs:ReceiveMessage, sqs:DeleteMessage, sqs:GetQueueAttributes, and permission to call the FastAPI service's VPC endpoint. Missing sqs:GetQueueAttributes causes a silent failure — Lambda won't tell you why it can't attach to the trigger; it just won't attach.

Building the OCR Pipeline

The OCR pipeline is the most technically interesting part of the system. Here's the full flow from a user clicking "Upload Receipt" to seeing extracted line items on screen.

  1. Upload initiation: The Next.js frontend calls POST /api/receipts/upload-url on Spring Boot. Spring Boot generates an S3 pre-signed URL (valid for 5 minutes) and returns it along with a new receiptId. The receipt row is created in RDS with status PENDING.
  2. Direct S3 upload: The browser uploads the image directly to S3 using the pre-signed URL. No image data passes through EC2.
  3. SQS enqueue: After the S3 upload succeeds, the frontend calls POST /api/receipts/{id}/process. Spring Boot enqueues a message: { "userId": "...", "s3Key": "...", "receiptId": "..." } and returns HTTP 202 Accepted immediately. API response time: under 200ms.
  4. Lambda consumption: Lambda polls the SQS queue and calls FastAPI's POST /extract endpoint with the S3 key. FastAPI downloads the image, runs the OpenCV pre-processing pipeline, executes Tesseract OCR, and returns structured JSON.
  5. Result persistence: Lambda calls Spring Boot's internal POST /internal/receipts/{id}/line-items endpoint to write the extracted items to RDS. The receipt status is updated to PROCESSED or FAILED.
  6. Frontend polling: The Next.js UI polls GET /api/receipts/{id}/status every 3 seconds until the status changes from PENDING. When it does, it fetches and renders the extracted line items.

Failed extractions are routed to a Dead Letter Queue (DLQ). A CloudWatch alarm fires if the DLQ depth exceeds zero, which triggers an SNS email notification.

How Claude 10x'd My Productivity

I kept a rough log of what Claude helped me build across this project. By the end I counted 14 distinct subsystems where I used it as a pair programmer — not a code generator I blindly copy-pasted from, but a collaborator I could interrogate, push back on, and ask "why" questions of. Here are the five that made the biggest difference.

1. Spring Boot JPA Entity Mappings

Malaysian tax reliefs have a complex nested structure: a parent relief category (e.g. "Medical") contains child relief items (e.g. "Medical examination fees"), each with year-specific claim caps and eligibility rules. Modelling this correctly in JPA — with the right cascade types, fetch strategies, and orphan removal settings — would have taken me a long-form dive into Hibernate docs.

Instead, I described the entity relationships to Claude and asked it to generate the full @OneToMany / @ManyToOne graph with validation annotations. It did, and it explained each cascade choice. What would have taken 45 minutes took 3. More importantly, I understood the output well enough to extend it when requirements changed.

2. FastAPI Pydantic Response Models

OCR output is inherently unpredictable. A receipt total might come back as "RM 45.00", "45.0", 45, or — for handwritten receipts — something completely malformed. Claude helped me design a Pydantic model with Optional fields, custom validator decorators that attempt multiple parsing strategies for amounts, and a confidence_score field that downstream code can use to decide whether to auto-accept or flag for manual review.

We iterated on this model three times as I encountered edge cases: handwritten receipts, dual-language receipts (Bahasa Malaysia / English), and receipts where the total was printed in a non-standard position. Each iteration took minutes, not hours.

3. The Lambda SQS Handler

Lambda's SQS integration has a subtle but critical behaviour: if your handler throws an exception, the entire batch is retried — including messages that succeeded. The correct pattern is to catch per-message failures and return them in a batchItemFailures array, so only the failed messages are retried.

Claude wrote the initial handler structure, including the retry logic and the batchItemFailures pattern, and explained why naive exception propagation causes duplicate processing. Here's the core of what it produced:

lambda_handler.py
import json
import logging

logger = logging.getLogger()
logger.setLevel(logging.INFO)

def handler(event, context):
    batch_item_failures = []

    for record in event["Records"]:
        receipt_id = None
        try:
            body = json.loads(record["body"])
            receipt_id = body["receiptId"]
            s3_key = body["s3Key"]

            # Download from S3 and run OCR via FastAPI
            result = ocr_service.extract(s3_key)

            # Write extracted line items back to RDS via Spring Boot
            spring_api.post_line_items(receipt_id, result)

        except Exception as e:
            logger.error(f"Failed to process receipt {receipt_id}: {e}")
            # Only failed messages go back to the queue — not the whole batch
            batch_item_failures.append(
                {"itemIdentifier": record["messageId"]}
            )

    return {"batchItemFailures": batch_item_failures}

4. Security Group Review — The Misconfiguration Claude Caught

I pasted my Security Group configuration into Claude and asked: "What's wrong with this from a least-privilege perspective?" It flagged something I'd missed: EC2 port 8080 was open to 0.0.0.0/0 — the entire internet — rather than scoped to the ALB's Security Group ID.

The fix

Change the inbound rule for port 8080 from source 0.0.0.0/0 to source sg-xxxxxxxx (your ALB's Security Group ID). This ensures that only traffic routed through the load balancer can reach the Spring Boot API — external scanners and bots cannot call it directly.

This was a real misconfiguration that would have exposed the Spring Boot API to the public internet. A security audit would have caught it eventually. Claude caught it in 30 seconds.

5. OpenCV Pre-processing Pipeline

Raw receipt photos are often blurry, skewed, or poorly lit. Running Tesseract directly on them produces error rates of 30–40% on low-quality inputs. After describing the problem and showing Claude a few example receipt descriptions, it suggested a 4-step pre-processing sequence:

  1. Grayscale conversion — removes colour noise that confuses character segmentation
  2. Gaussian blur — smooths minor sensor noise before thresholding
  3. Adaptive thresholding — handles uneven lighting across the receipt (a common problem with phone photos)
  4. Deskew via Hough transform — detects and corrects receipt tilt up to ±15°

Implementing this pipeline cut the unreadable-receipt rate from around 40% to approximately 15% on a test set of 200 real receipts. That's not a tuned production model — that's a 4-step OpenCV pipeline suggested in a single conversation.

Looking back at my time log: I estimate Claude saved approximately 60 hours over 3 weeks of evenings and weekends. But the hours saved is the less interesting metric. The more important effect is that Claude made me willing to attempt things I would have otherwise deprioritised.

Lambda + SQS from scratch, for a solo side project, on a deadline of "I want to ship this before LHDN filing season"? What I'd Do Differently

Shipping something and then learning what you should have done differently is how most of the best engineering judgment gets built. Here's mine.

  • ECS Fargate instead of raw EC2. Running Next.js and Spring Boot as co-located processes on a single EC2 instance works, but zero-downtime deploys require either a Blue/Green swap or careful PM2 orchestration. ECS Fargate with a rolling deployment strategy would handle this cleanly without manual SSH sessions.
  • An RDS read replica for reporting queries. The tax estimate dashboard runs aggregation queries across a user's full receipt history. As receipt counts grow, these queries compete with transactional writes on the same RDS instance. A read replica costs roughly double the RDS bill but eliminates this contention entirely.
  • CloudWatch alarms from day one. I added monitoring retroactively after the first time a Lambda function silently failed for 6 hours without me noticing. Alarms on DLQ depth, Lambda error rate, and RDS CPU should be part of the initial infrastructure setup, not an afterthought.
  • AWS CDK for IaC. All the infrastructure for this project was created through the AWS Console. That works for a side project, but it's unreproducible and undocumented. AWS CDK (with TypeScript) would let me define the entire stack in code, version-control it, and tear it down and rebuild it in minutes.

Key Takeaways

Three things I want you to walk away with from this post:

  1. Async decoupling is not premature optimisation. When your bottleneck is a 2–3 second ML call, making it synchronous is the premature decision. SQS + Lambda costs almost nothing at side-project scale and gives you horizontal scalability for free.
  2. Monorepo discipline pays off. Having all three services in one repository means cross-service changes — like updating the SQS message schema — are a single commit with a single review. Coordinated deploys become trivial. The friction of monorepo tooling setup is a one-time cost; the benefits compound.
  3. AI coding assistants change the effort curve, not just the speed. The real value isn't that Claude wrote code faster than I could. It's that it made the activation energy for hard problems low enough that I stopped deferring them. You tackle the right architecture instead of the expedient one.
